# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Development Commands

**Always use `make` targets instead of running Go commands directly.** The Makefile handles version injection, flags, and consistent tooling.

```sh
make build       # Build binary for current platform → ./jiru
make test        # Run tests with race detector (go test -race ./...)
make test VERBOSE=1  # Run tests with detailed output
make lint        # Run golangci-lint v2
make lint-fix    # Run golangci-lint with auto-fix
make check       # Run all checks: fmt, tidy, vet, lint, test
make build-all   # Cross-compile to dist/ (linux/darwin × amd64/arm64)
```

Version is injected at build time via `-X main.version=...` from `git describe`.

After completing a feature or fix, always run `make build` so the user can test the changes immediately via `./jiru`.

## Architecture

This is a terminal UI for Jira built with the [Bubble Tea](https://github.com/charmbracelet/bubbletea) framework (Elm architecture: Model → Update → View).

### Data flow

`main.go` → `config.MigrateToProfiles()` (migrates legacy `config.env` → `profiles.yml` on first run) → `config.PartialLoad()` (env vars → `~/.config/jiru/profiles.yml`) → if config complete, creates `client.Client` → passes to `ui.App` with partial config and missing fields. If fields are missing, the setup wizard is shown first.

### UI layer (`internal/ui/`)

- **`app.go`** — Root model. Contains `App` struct, `NewApp`, `Init`, `Update` (thin router dispatching to `handleKeyMsg`/`updateActiveView` + inline async message handlers), and `View`. Manages view states: `viewSetup` → `viewLoading` → `viewSprint` / `viewBoard` → `viewIssue` → `viewBranch`, plus `viewSearch` (overlay), `viewSearchBoard` (kanban board for search/filter results), `viewCreate` (issue creation wizard), `viewTransition` (status transition picker), `viewComment` (comment input), `viewFilters` (saved filter manager), `viewBoardPick` (board switcher), `viewSpaces` (Confluence space/page browser), and `viewConfluence` (Confluence page detail). Status messages display above the footer in green (success) or red (error) and auto-dismiss after 5 seconds. JQL search errors show as inline red status messages instead of the error overlay. Loading screen and setup wizard welcome step display the ASCII logo. When `BoardID == 0` after auth, the setup wizard is shown to configure a board.
- **`navigate.go`** — Navigation handler extracted from `app.go`. Contains `handleKeyMsg` (keyboard → view transitions, returns `(App, tea.Cmd, bool)` where `bool` indicates whether the key was consumed), `navigateBack` (back-navigation logic), `updateActiveView` (delegates messages to the active child view and handles sentinel-based navigation), `inputActive` (guards global keys when text input is focused), and `isBackKey`.
- **`commands.go`** — All `tea.Cmd` factory methods extracted from `app.go`: `verifyAuth`, `fetchSprintIssues`, `fetchMoreIssues`, `fetchIssueDetail`, `fetchChildIssues`, `fetchBoardsForPicker`, `fetchJQLMetadata`, `searchUsers`, `searchJQL`, `fetchTransitions`, `transitionIssue`, `addComment`, `toggleWatch`, `assignIssue`, `editIssue`, `fetchLinkTypes`, `linkIssue`, `deleteIssue`, `fetchIssueBundle`, `fetchBranchInfo`, `fetchActiveSprintForBoard`, `switchProfile`, Confluence fetch commands, plus utility functions (`openBrowser`, `copyToClipboard`, `sanitiseError`, `createBranch`, `clipboardBranch`, etc.).
- **`messages.go`** — All custom `tea.Msg` types (`ClientReadyMsg`, `SprintLoadedMsg`, `IssuesLoadedMsg`, `IssuesPageMsg`, `IssueSelectedMsg`, `IssueDetailMsg`, `OpenURLMsg`, `ErrMsg`, `BoardSelectedMsg`, `BoardPickLoadedMsg`, `SearchResultsMsg`, `SetupCompleteMsg`, `JQLMetadataMsg`, `UserSearchMsg`, `BranchCreatedMsg`, `IssueCreatedMsg`, `TransitionsLoadedMsg`, `IssueTransitionedMsg`, `CommentAddedMsg`, `ChildIssuesMsg`, `SpacesLoadedMsg`, `SpacePagesLoadedMsg`, `ConfluencePageMsg`). `IssuesLoadedMsg` and `IssuesPageMsg` carry pagination state (`HasMore`, `From`, `NextToken`, `Seq`) for progressive loading. `paginationSeq` on `App` prevents stale pages from corrupting views after navigation.
- **`keys.go`** — Global `KeyMap` with bindings (`esc` for back, `enter` for open, `s` for JQL search, `?` for help overlay, `/` for list filtering, `S` for setup, `n` for branch creation, `c` for create issue/comment, `m` for status transition, `f` for saved filters, `w` for watch/unwatch, `H` for home, `B` for board picker, `Tab` for Confluence wiki). `q` navigates back from sub-views and quits at the top level. Global keys are suppressed when text input is active (`inputActive()` guard in `navigate.go`). Note: `bubbles/list` default keybindings bind `f`/`d` to NextPage and `b`/`u` to PrevPage — these are overridden in issuelistview and searchview to avoid conflicts with app-level bindings.
- **`footer.go`** — Persistent keybind footer renderer. Context-sensitive bar showing relevant keybinds per view (replaces the old `?`-toggled help overlay).
- **`boardpickview/`** — Board switcher overlay. Async-loads boards from the Jira API, displays name + type. Triggered by `B` from sprint or board view. Exposes `Selected()` and `Dismissed()` sentinels.
- **`searchview/`** — JQL search with text input, results list, and context-aware autocomplete popup. Two states: `stateInput` (query entry with completion) and `stateResults` (browsable list). Uses `internal/jql` for parsing and completions — including dynamic values (statuses, issue types, priorities, etc.) from `ValueProvider` and live user search for assignee/reporter fields. `r` re-runs the current query from the results view. Exposes `SubmittedQuery()`, `SelectedIssue()`, `Dismissed()`, `SetMetadata()`, `SetUserResults()`, `NeedsUserSearch()`, and `SaveFilter()`.
- **`setupview/`** — Interactive setup wizard for first-run configuration. Walks through domain, user, API token, auth type, then presents picker steps for project and board (fetched from Jira API with ↑/↓ navigation), followed by git repo path, branch case, and branch mode toggles. Async API validation at credential checkpoint. Picker steps cache results and re-fetch when the project changes. Signals completion via `Done()` / `Quit()` sentinel fields. Displays the ASCII logo on the welcome step.
- **`branchview/`** — Branch creation wizard. Two text inputs (branch name + base branch) with autocomplete for local/remote branches. `Slugify()` converts issue key + summary to a branch slug — lowercase or title case with ALL-CAPS project key. `BranchRequest` carries name, base, repo path, mode (`local`/`remote`/`both`), and a `CopyName` flag that (when enabled via `BranchCopyName` config) tells `createBranch` to copy the branch name to the clipboard after a successful create. Exposes `SubmittedBranch()` and `Dismissed()` sentinels.
- **`transitionpickview/`** — Status transition picker overlay. Floating list of available transitions for an issue, with vim-style navigation. Triggered by `m` from issue or board view. Exposes `Selected()` and `Dismissed()` sentinels.
- **`commentview/`** — Comment input overlay. Multi-line textarea for posting comments using `bubbles/textarea`. Triggered by `c` from issue view. Submit with `ctrl+s`, cancel with `esc`. Exposes `SubmittedComment()` and `Dismissed()` sentinels.
- **`filterpickview/`** — Saved filter manager overlay. Lists saved JQL filters with cursor navigation, favourite toggling (`f`), edit (`e`), duplicate (`d`), copy JQL to clipboard (`x`), delete with confirmation (`D`), and new filter creation (`n`). Edit/create flows use a two-step form (name → JQL query) with JQL autocompletion via `internal/jql`. Filters are persisted to `~/.config/jiru/filters.json` via `internal/filters`. Exposes `Applied()`, `SaveRequested()`, `DeleteRequested()`, `FavouriteRequested()`, `DuplicateRequested()`, `CopyJQLRequested()`, and `Dismissed()` sentinels.
- **`createview/`** — Issue creation wizard. Multi-step form: project (picker) → issue type (picker, project-scoped) → summary (text input, required) → priority (picker) → assignee (text input with live user search) → labels (text input with autocomplete hints) → parent issue (text input) → description (text input) → confirm (summary view). Signals completion via `Done()` / `Quit()` / `CreatedKey()` sentinels. Triggered by `c` key from home, sprint, or board views. On success, navigates to the created issue detail view.
- **`issuedelegate/`** — Shared issue list delegate and `Item` type used by `issuelistview` and `searchview`. Renders two-line items (key + summary + status badge / type + assignee). `ToItems()` converts `[]jira.Issue` to `[]list.Item`.
- **`issuelistview/`** — Issue list using `bubbles/list` with the shared `issuedelegate.Delegate`. Supports `d`/`u` for half-page scrolling. `AppendIssues()` deduplicates by key and buffers items when a filter is active (flushed on filter clear) to avoid disrupting the user. `SetLoading()` supports progressive pagination. Exposes `SelectedIssue()` for parent to detect selection.
- **`boardview/`** — Kanban board view. Groups issues by status into columns, with card rendering and scrolling. Columns are sorted by status category (todo → in progress → done → cancelled) with workflow sub-priority within each category via `theme.StatusSubPriority()` (development before review before QA). Column navigation uses carry-with-memory: first visit carries the source cursor for visual continuity, return visits restore the column's saved position. Supports parent-based filtering (e.g., by Epic or Feature), `d`/`u` for half-page scrolling, and column windowing (capped at `maxVisibleColumns=4`, with `[n/total]` position indicator). `SetIssues()` populates the board; `AppendIssues()` deduplicates and preserves cursor positions during progressive pagination. `SetKnownStatuses()` accepts instance-wide status names (from JQL metadata) for proper column ordering. `b` toggles back to list view. When toggled from search/filter results, the board title shows the saved filter name (if applicable) via `searchBoardDisplayTitle()`.
- **`issueview/`** — Detail pane using `bubbles/viewport`. Renders metadata (status, type, priority, assignee, reporter, timestamps, labels), parent issue link (key + summary + type), child issues grouped by status category (To Do / In Progress / Done) with a progress bar, description (via `markup.Render`), and last 10 comments with wiki markup rendering and text wrapping. `SetChildren()` accepts `[]jira.ChildIssue` fetched asynchronously. Assignee, reporter, and comment authors use colour-coded `theme.UserStyle()`.
- **`wikilistview/`** — Confluence space/page browser using `bubbles/list`. Two states: `stateSpaces` (space list with recent pages) and `statePages` (pages in a selected space). `rebuildSpacesList()` orders global spaces before personal spaces, with recent pages at the top. Custom `spaceDelegate` renders two-line items (name + key badge / type + description). Exposes `SelectedPage()`, `SelectedItem()` (returns page ID for the highlighted item), `NeedsFetch()`, and `Dismissed()` sentinels. `Filtering()` / `Filtered()` delegate to the underlying list filter.
- **`wikiview/`** — Confluence page detail view using `bubbles/viewport`. Renders page title, metadata (space, author, version, timestamps), breadcrumb ancestors, and body content via ADF rendering (`internal/adf`). Supports page back-navigation stack. Exposes `CurrentPage()` for parent to extract page references and `CopyURL()` sentinel for copy-to-clipboard.

### Supporting packages

- **`internal/config/`** — Loads config from env vars (`JIRA_DOMAIN`, `JIRA_USER`, `JIRA_API_TOKEN`, `JIRA_AUTH_TYPE`, `JIRA_BOARD_ID`, `JIRA_PROJECT`, `JIRA_REPO_PATH`, `JIRA_BRANCH_UPPERCASE`, `JIRA_BRANCH_MODE`, `JIRA_BRANCH_COPY_NAME` — all centralised as unexported `env*` constants at the top of `config.go`), then `~/.config/jiru/profiles.yml` (profile-based config, written by setup wizard). `PartialLoad()` returns whatever values are available plus a list of missing required fields (used by setup wizard). `WriteConfigProfile()` persists settings to `profiles.yml` and stores the API token in the OS keychain via `keyring.go` (profile-aware keys). `MigrateToProfiles()` auto-migrates legacy `config.env` to `profiles.yml` on first run and deletes the old file. `JIRA_BOARD_ID` is now optional — when unset, the app shows the home screen with a board list. When `JIRA_API_TOKEN` is set via env var, the token is synced to the OS keychain so it persists across terminals. Supports aliases `JIRA_URL` and `JIRA_USERNAME`.
- **`internal/api/`** — Thin HTTP client for the Jira REST API. Handles authentication (Basic and Bearer), JSON encoding/decoding, and path helpers (`V1`, `V2`, `V3`) for different API versions. `DecodeResponse[T]` is a generic JSON decoder. No external Jira library dependency.
- **`internal/client/`** — Typed service methods (`Me`, `SprintIssues`, `SprintIssuesPage`, `GetIssue`, `Boards`, `BoardSprints`, `SearchJQL`, `SearchJQLPage`, `BoardIssuesPage`, `BoardFilterJQL`, `EpicIssues`, `EpicIssuesPage`, `SprintIssueStats`, `JQLMetadata`, `SearchUsers`, `Projects`, `Transitions`, `TransitionIssue`, `AddComment`, `ChildIssues`). Exports a `JiraClient` interface implemented by `*Client`, used by the UI layer for testability. `PageResult` type carries pagination state (`HasMore`, `From`, `NextToken`). The `*Page` methods return a single page; the non-paged methods loop internally for callers that need all results. `BoardFilterJQL` fetches a board's JQL filter via the board configuration API — used for boards without active sprints to avoid the Agile v1 truncation limit. JQL search uses the v3 `/search/jql` API with cursor-based pagination and explicit field list (not `*all`); sprint/epic use the Agile v1 API with offset-based pagination. `DefaultPageSize=100`, `MaxTotalIssues=2000`. Cursor loop detection breaks pagination if `nextPageToken` repeats (known Jira Cloud bug). `JQLMetadata()` makes parallel REST calls to fetch statuses (with category mapping from `statusCategory.key`), issue types, priorities, resolutions, projects, labels, components, and versions. `ChildIssues()` fetches child/subtask issues via JQL (`parent = KEY`). `jqlEscape()` prevents JQL injection in string literals. `convertIssue()` parses Created/Updated timestamps from the API.
- **`internal/validate/`** — Input validation helpers (`IssueKey`, `ProjectKey`, `Domain`, `Email`, `AuthType`, `BoardID`, `BranchName`) using regex. Used by `main.go` (CLI arg validation), `client` (JQL injection prevention), `branchview` (branch name validation), and `setupview` (wizard field validation).
- **`internal/jql/`** — JQL parser and autocomplete engine extracted from `searchview`. `Parse(query, cursor)` determines cursor context (field, operator, value, keyword, ORDER BY). `Match(ctx, provider)` returns completion candidates — including dynamic values from `ValueProvider`. `Accept()` inserts a completion. `RenderPopup()` renders the completion dropdown. Used by both `searchview` and `filterview`.
- **`internal/filters/`** — Saved JQL filter persistence. Stores filters as JSON in `~/.config/jiru/filters.json`. Provides `Load()`, `Add()`, `Update()`, `Delete()`, `Duplicate()`, `ToggleFavourite()`, and `Sorted()` (favourites first, then alphabetical). Each filter has an ID, name, JQL string, and favourite flag. `MaxFilterNameLen` (100) caps filter names and is shared with the UI input's CharLimit.
- **`internal/confluence/`** — Domain types (`Space`, `Page`, `PageSearchResult`, `PageAncestor`, `Comment`) for Confluence entities, decoupled from the API client.
- **`internal/adf/`** — Atlassian Document Format (ADF) renderer. `Render(json, width)` converts ADF JSON to styled terminal text using lipgloss. Handles paragraphs, headings, lists, code blocks, tables, panels, expand/collapse, media, inline marks (bold, italic, code, links, colours), and emoji. `ExtractPageRefs()` extracts Confluence page link references from ADF for cross-page navigation.
- **`internal/recents/`** — Recently viewed Confluence pages persistence. Stores entries as JSON in `~/.config/jiru/recents.json` (profile-aware via `SetProfile()`). `Add()` inserts or bumps a page to the top, trimming to `MaxEntries` (20). `Load()` / `Sorted()` for reading.
- **`internal/client/confluence.go`** — Confluence API methods on the shared `Client`: `Spaces()`, `SpacePages()`, `GetPage()`, `SearchPages()`. Uses the Confluence v2 REST API.
- **`internal/jira/`** — Domain types (`Issue`, `ChildIssue`, `Comment`, `Sprint`, `Project`, `Board`, `BoardStats`, `JQLMetadata`, `Transition`, `SavedFilter`) decoupled from the API client. `JQLMetadata.StatusCategories` maps status names to category ints (0=todo, 1=in progress, 2=done) from the Jira `/status` API. `Transition.ToStatus` carries the target status name (distinct from the transition action name).
- **`internal/markup/`** — Atlassian wiki markup renderer. `Render(input, width)` converts wiki markup to styled terminal text using lipgloss. Handles inline formatting (bold, italic, underline, strikethrough, monospace, links, images, colour), block elements (headings, lists, code blocks, noformat, panels, quotes, admonitions, tables, horizontal rules), mermaid diagram rendering (via `pgavlin/mermaid-ascii` — auto-detected in `{code}`, `{noformat}`, and `{mermaid}` blocks), and styled text wrapping. Opening tags with inline content and lenient closing tag detection are supported. Block-level elements (headings, lists) are rendered inside container blocks (panels, admonitions, quotes) via `renderContentLine()`.
- **`internal/theme/`** — Adaptive colours and lipgloss styles shared across views. `StatusStyle()` and `StatusCategory()` use the instance-specific status category mapping (set via `SetStatusCategoryMap()` from the `/status` API response) with hardcoded fallbacks for common names. `StatusSubPriority()` returns a workflow sub-priority for ordering statuses within the same category using keyword matching (development → review → QA); unrecognised statuses get a neutral value (50). `UserStyle()` returns a deterministic colour for a user name (consistent hash into a 10-colour palette). `RenderLogo()` returns the ASCII art logo styled in muted blue (or empty if the terminal is too narrow).

### UI styling conventions

Dialogs and overlays follow a consistent style pattern using `internal/theme` colours:

- **Primary dialogs** (confirmations, setup wizard) — `RoundedBorder()` with `ColourPrimary` (blue) border, `Padding(1, 3)`, centred via `lipgloss.Place()`. Title text uses bold + `ColourPrimary`.
- **Error dialogs** — `StyleErrorDialog`: `RoundedBorder()` with `ColourError` (red) border, same padding and centering.
- **Info panels** (admonitions) — `RoundedBorder()` with colour matching the macro type (blue for info, yellow for warning, green for tip, subtle for note).
- **Status messages** — displayed above the footer: green (`ColourSuccess`) for success, red (`ColourError`) for errors. Cleared on the next keypress.
- **Subtle/secondary text** — `ColourSubtle` for hints, descriptions, and less important UI text.

When adding new dialogs or overlays, follow the primary dialog pattern (blue border, centred, padded) unless it's an error or warning.

### Jira API constraints

- **v2 `/rest/api/2/search` is gone** — Atlassian removed it (410 Gone) as of May 2025. JQL search must use v3 `/rest/api/3/search/jql` with cursor-based `nextPageToken` pagination. Do not introduce new v2 search calls.
- **Agile v1** (`/rest/agile/1.0/`) endpoints (sprints, boards, epics) still use offset-based `startAt` pagination. The `IsLast` flag is unreliable on Jira Cloud — pagination continues until an empty page is returned (with `MaxTotalIssues` as a safety cap).
- **v2 REST** (`/rest/api/2/`) is still used for non-search operations: issue CRUD, transitions, statuses, metadata, comments, etc.

### Key pattern

Child models (`boardlistview.Model`, `issuelistview.Model`, `boardview.Model`, `issueview.Model`, `searchview.Model`, `setupview.Model`, `transitionview.Model`, `commentview.Model`, `filterview.Model`, `wikilistview.Model`, `wikiview.Model`) are value types. They signal events to the parent via sentinel fields (e.g., `SelectedBoard()`, `SelectedIssue()`, `SubmittedQuery()`, `OpenURL()`) rather than returning messages — the parent polls these after calling `Update`.
